En esta directriz se describen cuestiones importantes a tener en cuenta cuando se personalizan los aspectos de Gestión de proyectos del proceso.

Descripción principal

Decidir cómo utilizar los productos de trabajo

Decida qué productos de trabajo se utilizarán y cómo se utilizarán. Además de identificar qué productos de trabajo se utilizarán, también es importante personalizar cada producto de trabajo que se utiliza para que se ajuste a las necesidades del proyecto. 

La tabla siguiente especifica qué productos de trabajo de gestión del proyecto se recomiendan y cuáles se consideran opcionales (por eje., sólo se pueden utilizar en algunos casos). Para consideraciones adicionales sobre la personalización, consulte la sección de personalización de la página de descripción de los productos de trabajo.

Producto de trabajo Objetivo

Personalización (Opcional, recomendada)

Caso de negocio Se utiliza para determinar si vale la pena invertir en el proyecto.

Recomendada.

Valoración de iteración Captura el resultado de una iteración, el grado hasta el cual se cumplen los criterios de evaluación, las lecciones aprendidas y los cambios que se deben realizar.

Recomendada.

Plan de iteración Plan detallado de la iteración, incluida la secuencia de tiempo de las tareas y los recursos. Recomendada.

Plan de desarrollo de software

  • incluidos, por lo general:
  • Plan de medida
  • Plan de resolución de problema
  • Plan de aceptación de producto
  • Plan de garantía de calidad
  • Plan de gestión de riesgos
Incluye toda la información que se necesita para gestionar el proyecto.

Todos los proyectos necesitan alguna planificación para gestionar un proyecto.

Los proyectos más pequeños y menos complejos pueden tener un único documento que capture el plan del proyecto. Los proyectos más grandes, complejos y formales requieren varios subplanes separados.

Medidas del proyecto Depósito de todas las medidas relacionadas con el proyecto.

Recomendado para la mayoría de proyectos.

En muchos proyectos, sólo se utilizan unas pocas medidas como, por ejemplo, medidas de coste y progreso. Sólo se requiere una base de datos de métrica cuando se debe gestionar una gran cantidad de datos de métrica. Muchas organizaciones recopilan datos de métrica de varios proyectos con el objeto de recoger información que aplicar a futuros proyectos.

Registro de revisión

Captura los resultados de una revisión de uno o más productos de trabajo de proyectos.

Los registros de revisión pueden evitar equivocaciones de decisiones tomadas durante una revisión. También sirven como pruebas para los interesados de que se han revisado los productos de trabajo de proyectos.

Recomendado para la mayoría de proyectos.

En la mayoría de proyectos se desea registrar las decisiones tomadas en reuniones con el cliente y otros interesados clave con el objeto de asegurar un entendimiento común.

Los registros de revisiones de otras revisiones se pueden capturar formalmente, en función de la formalidad de revisión que aplica el proyecto concreto.

Lista de riesgos Es una lista del orden de prioridades de los riesgos de proyecto.

Recomendada.

Puede ser, simplemente, una sección del plan de desarrollo de software.

Valoración de estado Se utiliza para capturar una instantánea del estado del proyecto, incluido el progreso, problemas de gestión, problemas técnicos y riesgos.

Recomendada.

La Valoración de estado se puede combinar con la Valoración de iteración, si las iteraciones son frecuentes (una cada mes). Si las iteraciones se producen con menor frecuencia, se deben hacer Valoraciones de estado intermedias.

Pedido de trabajo Es un acuerdo negociado entre el gestor de proyectos y el personal para efectuar una tarea concreta, o un conjunto de tareas, bajo una planificación definida y con ciertos productos de trabajo, esfuerzo y restricciones de recursos.

Recomendado para la mayoría de proyectos.
