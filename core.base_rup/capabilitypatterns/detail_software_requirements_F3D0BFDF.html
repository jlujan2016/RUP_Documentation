Tarea: Detallar los requisitos de software
En esta tarea se describe cómo detallar los requisitos de software del sistema.

El plan de gestión de requisitos define la ubicación y organización de los requisitos y, por tanto, afecta directamente a cómo y dónde se detallan los requisitos.
Objetivo

El objetivo de esta tarea es recopilar, detallar y organizar el conjunto (paquete) de productos de trabajo que describen completamente los requisitos de software del sistema.

Descripción principal
Detallar los requisitos de software implica detallar los guiones de uso y las especificaciones suplementarias de los requisitos que se van a proporcionar en el release actual.   Para obtener más información, consulte Detallar un guión de uso y Desarrollar especificaciones suplementarias.
Pasos
Detallar los requisitos de software

Asegúrese de que se especifican todos los requisitos con el nivel de detalle necesario para distribuirlos a los diseñadores, verificadores y escritores de documentación. Revise las listas de comprobación asociadas con los productos de trabajo que se están produciendo.

Si los requisitos se rastrean o se gestionan formalmente de otro modo, asegúrese de que cada requisito esté bien identificado y etiquetado.

Generar informes de soporte

Los requisitos a menudo se almacenan y se gestionan utilizando una o más herramientas. Por ejemplo, herramientas de:

  • modelado gráfico de requisitos
  • gestión de requisitos generales y de rastreabilidad
  • otra documentación textual y gráfica.

En este paso se genera documentación a partir de estas herramientas, para que se pueda revisar la información fácilmente. Si no se utilizan herramientas especializadas para capturar los requisitos, este paso no se aplica (todos los requisitos de software se escribirán directamente en la documentación).

Para obtener información sobre los informes que dan soporte a la producción de los productos de trabajo, consulte las páginas de contenido de los productos de trabajo.

Empaquetar los requisitos para su revisión

En este paso es donde se produce una definición completa de los requisitos de software que se van a entregar en el ciclo de desarrollo actual para que los requisitos se puedan recuperar de manera eficaz.  Todos los productos de trabajo de requisitos se ensamblan y se proporciona información adicional.   

Para los proyectos menos formales, este paso consiste en empaquetar los informes relevantes y la documentación generada a mano, con material de soporte suficiente.

En los proyectos más formales, se pueden producir una o varias Especificaciones de requisitos de software (SRS) que recopilan y organizan todos los requisitos que envuelven el proyecto. Por ejemplo, un SRS separado puede describir los requisitos de software completos para cada característica en un release determinado del producto. Esto puede incluir los requisitos que se documentan en otros productos de trabajo de requisitos.  En tales casos, el SRS puede hacer referencia a los requisitos documentados en los otros productos de trabajo.  Esto permite tener un SRS formal sin necesidad de duplicar la información en los otros productos. Utilizando la plantilla de Especificación de requisitos de software adecuada, ensamble las partes del paquete SRS y proporcione el resto de información para tener una definición completa de los requisitos de software de este subsistema o esta característica.

Factores clave

El Prototipo de interfaz de usuario puede ser una excelente fuente de requisitos detallados que se han podido omitir durante la obtención de requisitos inicial.   Si el usuario acepta el prototipo, es importante documentar de forma explícita los requisitos detallados necesarios para implementar el prototipo.
