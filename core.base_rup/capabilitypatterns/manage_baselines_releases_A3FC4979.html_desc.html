Actividad: Gestionar líneas base y releases
Esta actividad garantiza que conjuntos coherentes de artefactos relacionados o dependientes puedan identificarse como parte de una "línea base" con distintas finalidades, como la identificación de los posibles releases, versiones del producto, madurez del artefacto o completitud.

Descripción

La frecuencia y la formalidad con la que se crean las líneas base se describe en el plan de CM. El grado de formalidad es claramente mucho mayor para un producto que se va a entregar a un cliente que el de los releases ejecutables que se produzcan internamente para el equipo del proyecto. Cuando el conjunto combinado de artefactos alcanza ciertos niveles de madurez, se crean líneas base para ayudar a gestionar su disponibilidad para la publicación, la reutilización, etc.

Este trabajo suele dirigirlo fundamentalmente el rol de gestor de configuración, en el que la necesidad habitual es poder ensamblar un producto para publicarlo. El producto publicado necesita una lista de materiales (bill of materials, BOM) que se utiliza como lista de comprobación completa de lo que debe entregarse al cliente. El producto publicado precisará inevitablemente de unas notas del release y de material de formación, según se describe en las actividades de despliegue.

El rol de integrador contribuye a este trabajo garantizando que los artefactos entregados desde los espacios de trabajo de desarrollo se hayan integrado para que puedan incorporarse a una compilación que se pueda probar de forma independiente. La persona que desempeñe el rol de integrador debe estar familiarizado con las políticas y las prácticas de prueba de CM del proyecto.

Utilización
Instrucciones de utilización

Normalmente, este trabajo se inicia en la elaboración, aunque es más predominante en las fases de construcción y de transición. En algunos casos, puede valorarse la línea base de artefactos de la fase inicial, pero esto no es la norma general. Pueden crearse y gestionarse varias líneas base en una sola iteración y puede tener lugar en cualquier momento de la iteración.

Como norma general, este trabajo es opcional si no es necesario entregar externamente un release ejecutable fuera del equipo de desarrollo principal.
