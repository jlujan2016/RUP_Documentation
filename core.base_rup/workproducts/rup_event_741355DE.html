Artefacto: Suceso
Este producto de trabajo especifica una aparición en espacio y tiempo; menos formalmente, una aparición de algo a lo que el sistema debe responder.
Clases de producto de trabajo: Elemento de modelo
Objetivo

El objetivo de este producto de trabajo es capturar las características de los sucesos, como la frecuencia, la prioridad y los requisitos de respuesta.

Descripción
Descripción principal

Un suceso se utiliza para identificar y capturar información sobre apariciones externas de las que el sistema se da cuenta y a las que debe responder. Los sucesos también se pueden utilizar para capturar información sobre sucesos internos, como las excepciones.

Esquematización breve

Las características más importantes de los sucesos son:

  • interno o externo - ¿El suceso es externo o interno?
  • prioridad - ¿Este suceso debe provocar la suspensión de otros procesos para poder manejarlo?
  • frecuencia - ¿Con qué frecuencia se produce el suceso?
  • distribución de frecuencia - ¿Se produce el suceso a intervalos regulares o hay picos?
  • requisitos de respuesta - ¿Con qué rapidez debe responder el sistema al suceso (distíngase entre la media y el peor de los casos)?
  • tipo - ¿Es un suceso de llamada, un suceso de tiempo, un suceso de señal o un suceso de cambio (consulte Concepto: Sucesos y señales para ver las definiciones)?
Personalización
Opciones de representación

En el contexto de los diagramas de estado y de actividad, Suceso hace referencia a un desencadenante para una transición de estado.

No obstante, este producto de trabajo cubre "suceso" en el sentido más general, como ocurrencias a las que debe responder el sistema, como señales, llamadas, cambios de estado o sucesos de tiempo.

Consulte también el apartado Producto de trabajo: Señal.

Las características de sucesos se pueden capturar en una hoja de cálculo, una base de datos, una base de datos de gestión de requisitos o en forma de tabla en el documento de arquitectura de software.

Incluso se pueden capturar como clases, estereotipadas como <<suceso>>, aunque este método debería tratarse como un modo adecuado de capturar información de gestión sobre sucesos y no confundirlo con los datos transmitidos cuando se produce el suceso. Si una llamada de suceso da como resultado la transmisión de datos, los datos deben representarse mediante la firma de la operación a la que se ha llamado. Si el suceso es una señal, sus datos se pueden modelar explícitamente (consulte el apartado Producto de trabajo: Señal).
